## 3. iOS/Android 코드 검수 후 제외된 것

### ❌ 감정 타임라인 (River) → 제외

**이유**: iOS에 이미 5개 차트 탭(흐름 FlowChartView / 월별 MonthlyChartView / 분포 MoodDistributionView / 날씨 WeatherStatsView / AI분석 ReportView)이 구현되어 있음. Android도 StatsScreen.kt에서 동일 구현. "더 예쁜 차트"는 킬러 피처가 아니라 디자인 개선.

### ❌ Apple Watch 체크인 → 후순위

**이유**: Xcode 프로젝트에 watchOS Target 자체가 없음. 완전 새 플랫폼 확장이라 구현 기간이 길고(1~2주+), Apple Watch 보유율 제한.

### ❌ 마음 날씨 예보 → 대표가 직접 거부

**대표 원문**: "마음 날씨 예보는 내가 왜 안했을거 같아?!?!!?!"

**안 한 이유 (AI 반성)**:

- `condition.py` 코드 주석에 명시: **"틀려도 사용자에게 해가 없는 메시지만 사용한다."**
- **자기충족적 예언(Self-fulfilling prophecy)**: "내일 흐림"이라고 하면 진짜 우울해질 수 있음
- **예측 실패 시 불안 유발**: 멀쩡한데 "흐림 예보"를 보면 불필요한 걱정
- **임상 윤리 위반**: B2G(보건소) 연동 앱에서 근거 없는 미래 감정 예측은 위험
- **감정은 날씨가 아님**: 예측 불가능한 외부 변수(사건, 사람, 환경)에 의해 순간적으로 바뀜
- 7일치 mood trend로 "내일 감정"을 예측하는 건 **과학이 아니라 점술**

→ AI가 임상적 리스크를 고려하지 않은 **우선순위 판단 오류(Prioritization Failure)**였음을 인정.

## 4. 최종 남은 후보 2개 — 상세 분석

### 🥇 AI 주간 편지

**컨셉**: 매주(일요일 등) AI가 한 주를 돌아보며 따뜻한 편지를 써줌. Push 알림으로 도착.

**기존 AI 리포트와의 차이점**:

|             | 기존 AI 리포트 (ReportView) | AI 주간 편지                  |
| :---------- | :-------------------------- | :---------------------------- |
| 트리거      | 사용자가 버튼 클릭 (수동)   | 앱이 알아서 보냄 (능동)       |
| 데이터 소스 | 일기 raw 텍스트             | Kick Phase 1+2+3 인사이트     |
| 톤          | 분석적 ("3줄 요약")         | 감성적 ("편지")               |
| 개인화 깊이 | 낮음 (일기 내용만)          | 높음 (행동패턴+언어변화+관계) |
| UX          | 텍스트 블록                 | 봉투 열기 애니메이션          |

**경쟁사 대비**:

- 마인디: "오늘 일기에 답장" → 즉시 반응, 기억력 없음
- 마음온 편지: "일주일을 지켜본 뒤 편지" → 시간을 두고, 패턴을 읽고, 종합적으로

**마음온만 가능한 이유**: Kick 3Phase 분석 파이프라인이 없으면 아래 수준의 편지를 쓸 수 없음:

```
"이번 주에 엄마와의 시간이 늘었고, 그때마다 기분이 올라갔어요 💛
 반면 수요일부터 문장이 짧아지고 어휘가 줄었어요.
 혹시 피곤하신 건 아닌지 걱정이 되네요.
 이번 주말엔 좋아하는 산책로를 걸어보는 건 어떨까요? 🌿"
→ Phase 1(빈도+무드트렌드) + Phase 2(문장길이+어휘변화) + Phase 3(엄마=기쁨) 통합
```

**리텐션 구조**:

- 주 1회 Push: "💌 이번 주 마음 편지가 도착했어요"
- 편지 열기 → 앱 진입 → 캘린더 → "오늘도 써볼까?" → 일기 작성

**리스크**:

- LLM 품질이 떨어지면 감동 없음 → 생성 품질이 생명
- 일기를 안 쓴 주에는 내용 빈약 → 빈 주 대응 로직 필요
- 매주 비슷한 톤이면 질림 → 문체/톤 변화 로직 필요

**판단**: 조건부 YES. Kick 인사이트가 편지에 자연스럽게 녹아들어야만 킥. "그냥 따뜻한 말"이면 킥 아님.

### 🥈 관계 지도 (인터랙티브 소셜 맵)

**컨셉**: 일기에 등장하는 인물들을 네트워크 그래프로 시각화. 노드 크기 = 언급 빈도, 색깔 = 감정.

**현재 구현 상태**:

- **백엔드**: ✅ 100% 완료 (`relational.py` — Hybrid NER, 인물 추출, 감정 매핑, `kick_routes.py` API 존재)
- **의료진 대시보드**: ✅ 데이터 표시됨
- **사용자용 프론트 (iOS/Android)**: ❌ 전혀 없음

**킥 가능성**:

- "나도 몰랐던 나의 패턴"을 눈으로 보여줌 → 자기발견
- "엄마를 15번 언급, 그때마다 기뻤다" → 스스로 인지 못하는 것을 시각화
- 스크린샷 하나로 앱스토어에서 눈길 → "감정 차트"는 다 있지만 "관계 감정 네트워크"는 본 적 없음

**마케팅 한 줄**: "당신이 쓴 일기에서 사람들을 찾아내고, 그 사람과의 감정을 별자리처럼 그려드려요."

**리스크**:

- 데이터 없으면 빈 화면 → 최소 7~10일, 3명 이상 인물 언급 필요. 그 전엔 잠금
- 부정적 관계 표현 → "분노" 빨간 노드 = 자극적 → 부드러운 표현 필수
- "한번 보고 끝" 위험 → 시간에 따른 변화를 보여줘야 지속 관심

**판단**: 단독으로는 불충분. AI 편지와 결합하면 킥.

## 8. AI의 킥(Kick) 방어 및 추가 제안 (260305 08:08)

대표님께서 "아직도 킥이 될 만한 기능을 못 찾은 것 같다"고 느끼시는 것은 매우 합리적인 직관입니다.

단순히 "AI가 편지를 써준다"거나 "관계도를 그려준다"는 개념 자체는 다른 앱(마인디 등)이나 기존의 멘탈 케어 서비스들이 주장하는 기능들과 문장 레벨에서는 크게 다를 바 없어 보이기 때문입니다. 즉, 기능 설명만 들었을 때 **"이것 때문에 당장 앱을 깔고 싶다(바이럴)"** 또는 **"이거 확인 안 하면 찝찝하다(리텐션)"**라는 파괴력이 확 와닿지 않는 상태입니다.

말씀하신 대로, 현재 남아있는 두 가지 (AI 주간 편지, 관계 지도 시각화)가 **어떻게 '진정한 킥(Killer Feature)'으로 작동할 수 있는지**에 대한 구체적인 시나리오와 논리를 증명해보겠습니다. 나아가, 그럼에도 부족하다면 Kick의 Phase 1~3 엔진을 활용해 방향을 완전히 비튼 새로운 킥 아이디어도 함께 제시하겠습니다.

### 1. 두 기능이 결합했을 때 '킥'이 되는 논리적 이유

이 두 가지는 개별 UI로 존재하면 단순한 부가 기능(Nice-to-have)이지만, **완벽히 융합되어 하나의 UX로 제공될 때(Storytelling)** 타 서비스가 기술적으로 흉내 낼 수 없는 해자가 생깁니다. 핵심은 **마음온 Kick의 Phase 2(언어 지문)와 Phase 3(관계 지형도)**의 연계입니다.

- **타사의 한계 (1차원적 분석):** "이번 주 우울한 날이 많았네요. 힘내세요." (로그인 빈도, 단순 감정 태그 기반)
- **마음온의 타격감 (3차원적 인지):**
  > _"이번 주 당신의 우주에서 **'팀장님'**이라는 별이 가장 붉게(스트레스) 타올랐네요. 그래서인지 수요일부터 문장 길이가 평소보다 30%나 짧아지고 방어적인 어휘가 늘어난 것이 보입니다. 하지만 그 옆의 조그만 **'강아지'**라는 별이 가장 따뜻한 빛을 내며 당신을 지켜주고 있었어요."_

**이것이 왜 킥(Kick)인가?**

1.  **시각적 와우 포인트 (Visual WOW):** 단순히 텍스트 리포트를 읽는 것이 아니라, 앱을 켜면 내 인간관계가 우주나 별자리(관계 지도)로 펼쳐져 있고, 그 위를 AI 편지가 내레이션 하듯 짚어줍니다.
2.  **공유 욕구 (Biability/Viral):** "내 인간관계 별자리"라는 이미지는 인스타그램 스토리에 매우 공유하기 좋은 포맷입니다. (B2C 확장의 핵심)
3.  **메타 인지 충격:** 사용자는 무의식적으로 쓴 일기에서 "내가 요새 말이 짧아졌다는 것"이나 "특정 인물에게 감정 에너지를 다 뺏기고 있다는 것"을 AI가 정확히 짚어낼 때 소름을 느낍니다. 이는 높은 잔존율(Retention)로 이어집니다.

### 2. (추가 제안) 그래도 부족하다면? 방향을 비튼 '진짜 찐 타격감' 후보

위의 내러티브적 킥으로도 파괴력이 약하다고 판단되신다면, 현재 개발된 Kick 엔진(특히 Phase 3 인물/관계 추출)을 극한으로 활용한 **날카로운 기능 3가지**를 추가 제안합니다.

#### 💡 가설 A. "감정 가계부 (Mind Energy Ledger)"

돈을 어디에 썼는지 가계부를 적듯, **나의 정신적 에너지(감정)를 누구/무엇에 가장 많이 소비했는지 재무제표 형태**로 보여주는 기능입니다.

- **원리:** Phase 3(관계 추출) + Phase 1(감정 트렌드) 결합.
- **보여주기 방식:** "이번 주 감정 지출 1위: **회사 (에너지의 65% 소모, 적자)** / 감정 충전 1위: **산책 (20% 충전)**"
- **킥 포인트:** 막연한 감정을 '에너지 잔고'라는 구체적인 숫자로 치환하여 직관성을 극대화합니다. 바쁜 직장인들에게 "아, 내가 이번 주에 쓸데없는 곳에 멘탈 에너지를 너무 썼네. 주말엔 아껴야겠다"라는 실질적인 통제를 가능하게 합니다.

#### 💡 가설 B. "마음의 스파이 (나를 해치는/살리는 숨은 단어 찾기)"

단순 나열이 아니라, 사용자의 일기 데이터(Phase 2, 3)를 분석하여 **"나를 우울하게 만드는 트리거 단어/사람"과 "나를 회복시키는 단어/사람"을 AI가 발굴해서 직언**하는 기능입니다.

- **원리:** 특정 인물이나 상황이 언급된 날의 감정 점수 상관관계를 수치화.
- **보여주기 방식:** "분석 결과, [수면 부족]이라는 단어가 등장한 날, 당신의 긍정 수치는 다음날까지 40% 하락합니다." 또는 "당신은 [동기 이름]와 함께 있을 때 사용하는 단어의 폭이 평소보다 2배 넓어지며 긍정적인 자아를 보입니다."
- **킥 포인트:** AI가 무의식 속에 숨겨진 "내 기분의 스위치"를 찾아준다는 점에서 매우 강력한 컨설팅 도구로 느껴집니다.

#### 💡 가설 C. "과거와의 조우 (Time-Capsule Rescue)"

사용자의 일기가 어느 정도 쌓인 시점(1개월 이상)에서 발동하는 킥입니다.

- **원리:** 과거 가장 부정적인 감정(Score)을 기록했던 날의 일기 데이터를 찾아, 현재 긍정적인 날의 사용자에게 띄워주며 AI가 두 시점을 이어줌.
- **보여주기 방식:** _"3주 전 수요일, 당신은 '모든 게 무너진 것 같다'고 적었지만, 오늘 당신은 '점심이 맛있었다'고 적을 만큼 단단해졌습니다. 그때의 당신에게 오늘 어떤 말을 해주고 싶나요?"_
- **킥 포인트:** 멘탈 케어의 핵심인 '회복 탄력성(Resilience)'을 가장 극적으로 체험하게 해줍니다. 기술 구현(데이터 조회 + 프롬프트 처리)은 매우 간단하지만, 사용자가 느끼는 감동과 눈물 버튼(Emotional Trigger)은 압도적입니다.

## 10. AI의 아키텍처 및 비용 검증 (260305 08:14)

대표님께서 비즈니스와 인프라의 최고 핵심(Bottleneck)을 정확하게 찔러주셨습니다. 좋은 기획이라도 런타임 비용이 감당 안 되거나 서버가 터진다면 그건 '킥'이 아니라 '독'입니다.

결론부터 말씀드리면, **비싼 외부 API(OpenAI 등)나 추가 서버 증설 없이 현재 우리 인프라(OCI 4C/24GB + 로컬 앱 렌더링)만으로 충분히 구현 가능합니다.** 그 이유는 이 기능들이 요구하는 연산의 성격이 "실시간 채팅"과 다르기 때문입니다.

### 1) AI 주간 편지 👉 **OCI 서버 (백그라운드 배치 작업)**

- **작동 방식:** 앱을 켜고 기다리는 것이 아닙니다. 매주 일요일 새벽(또는 트래픽이 적은 고정 시간)에 **OCI 서버가 알아서 주기적으로 작업(Cron Job)**을 시작합니다.
- **인프라 감당 여부:** 🟢 **가능.** OCI(4코어/24GB)에서 `Gemma-2-2b` 모델을 통해 편지 한 통을 쓰는 데(CPU 추론) 약 10~20초가 걸린다고 가정해도, 사용자가 수천 명일 때 서버가 큐(Queue)에 넣어두고 순차적으로 비동기 처리하여 편지만 DB에 미리 쌓아둡니다.
- **로컬 LLM이 불가능한 이유:** iOS/Android의 백그라운드 구동 제한 정책(Background Fetch 등) 때문에, 앱이 완전히 꺼져있을 때 무거운 로컬 LLM을 몰래 구동하여 편지를 작성하게 할 수 없습니다. 즉, 주간 편지는 "서버에서 만들어두고 Push 알림으로 날려주는 것"이 유일한 정답입니다.

### 2) 관계 지도 / 감정 가계부 👉 **기존 서버 로직 + UI 렌더링 (LLM 비중 최소화)**

- **작동 방식:** 이 기능들은 무거운 언어모델 연산이라기보다 **데이터 시각화**에 가깝습니다. 일기를 쓸 때 이미 백그라운드에서 분해된 Phase 3 (인물/상황) 데이터를 **통계(SQL 쿼리)**로 가져옵니다.
- **인프라 감당 여부:** 🟢 **완전 가능.** "엄마를 이번 주에 5번 썼고 평균 감정 스코어는 80점"이라는 계산은 단순 프로그래밍 영역이므로 서버 CPU 소모는 0.01초 수준입니다. 디바이스(앱)는 LLM 연산 없이 그 가벼운 JSON 결과값만 받아서 멋진 UI(별자리 노드, 가계부 차트)로 **그려주기(Rendering)**만 하면 끝납니다.

### 3) 과거와의 조우 / 마음의 스파이 👉 **OCI 서버 캐싱 + 부분 로컬LLM 하이브리드**

- **작동 방식:** 서버가 DB에서 "한 달 전 가장 우울했던 날의 일기"를 찾아냅니다. 그리고 두 시점을 이어주는 짧은 코멘트를 붙입니다.
- **인프라 감당 여부:** 🟢 **가능.** 멘트가 매우 짧아 OCI 서버에서 생성해도 부하가 없으며, 향후 비용(서버 점유 시간)을 아예 0으로 하고 싶다면 이 파트의 프롬프트만 앱에 내장된 **로컬 LLM(iOS/AOS)**에게 짬처리(Offloading)하여 사용자 폰에서 실시간으로 멘트를 붙이도록 듀얼 아키텍처로 짜도 됩니다.

### 🔥 최종 분석 요약 (가장 중요한 포인트)

해당 아이디어들이 억지 기획이 아니라 진짜 '가성비 킥'이 될 자격이 있는 이유는 **"실시간 생성 대기(Synchronous Loading)를 아예 회피한 설계"**이기 때문입니다.

사용자가 "분석해줘!"를 누르고 뱅글뱅글 도는 로딩 화면을 기다리는 방식이면 OCI 4코어로는 동시접속자가 몰렸을 때 뻗어버릴 수 있습니다.
하지만 위 제안들은 **"사용자가 모르는 사이 서버가 자원적 여유가 있을 때(비동기/배치) 미리 분석을 끝내놓고, 결과만 데이터베이스에 캐싱해두었다가 앱에서는 화려한 UI로 꺼내 쓰는 방식(Pre-computed)"**입니다. 현재의 무상 OCI 24기가 램과 4코어 환경을 극한까지 뽑아 먹으며 대규모 상용 서비스의 퍼포먼스를 내는 완벽한 전략입니다.
