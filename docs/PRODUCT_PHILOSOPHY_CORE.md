# 마음온(Maum-On) 제품 철학 및 기능 제안 가이드라인

> **작성일**: 2026-03-03
> **목적**: 마음온에 새로운 기능을 제안하거나 검토할 때 반드시 참고해야 할 제품 철학과 의사결정 원칙을 정리한다.
> **이 문서는 모든 기능 제안 전에 반드시 읽어야 한다.**

---

## 1. 핵심 원칙: 가치 사슬(Value Chain)을 이해하라

마음온의 모든 가치는 아래 흐름에서 파생된다:

```
사용자가 자신의 생각을 글로 쓴다 (일기)
           ↓
On-Device AI가 텍스트를 분석한다 (감정, 키워드, 뉘앙스)
           ↓
마음 온도(Mind Temperature)가 산출된다
           ↓
의료진 대시보드에 임상 인사이트가 표시된다
           ↓
위기 감지 → 보호자/의료진 알림 → 개입
```

### ⚠️ 절대 원칙

> **이 가치 사슬의 원점 — "사용자가 자신의 생각을 글로 쓴다" — 을 훼손하는 기능은 절대 제안하지 마라.**

사용자의 텍스트가 없으면:

- AI가 분석할 것이 없다
- 마음 온도를 산출할 근거가 없다
- 의료진 대시보드에 보여줄 인사이트가 없다
- 위기 키워드 감지가 작동하지 않는다

**텍스트 입력은 마음온의 존재 이유 그 자체다.**

## 2. 금지 사항: "편의성"이라는 명목의 가치 파괴

### ❌ 이모지 1탭 기록

```
😢 탭 → 끝.
AI가 분석할 텍스트: 없음.
마음 온도 산출 근거: mood_score 숫자 1개뿐.
의료진이 받는 인사이트: "환자가 슬프다고 했습니다." (이게 전부)
```

**왜 안 되는가**: 이모지는 사용자의 **생각**을 담지 못한다. "왜 슬픈지", "어떤 상황인지", "얼마나 심각한지"를 AI가 알 수 없다. 이건 "초저에너지 입력"이 아니라 **가치 사슬의 원점을 절단하는 것**이다.

### ❌ 예/아니오 초간단 응답

```
AI: "오늘 힘드셨나요?"
사용자: "네" → 끝.
AI가 분석할 텍스트: "네" (1글자)
```

**왜 안 되는가**: 마음온의 AI 선제 질문은 사용자에게서 **서술형 답변을 이끌어내기 위한 수단**이다. 예/아니오로 끝나면 질문의 존재 이유가 사라진다. 기존 Diary Interview Sequence(Event → Emotion → Reflection → Validation)는 **텍스트를 더 잘 뽑아내기 위해** 설계된 것이지, 텍스트를 대체하기 위한 것이 아니다.

### 왜 이런 실수가 발생하는가

이해관계자 분석에서 우울증 환자가 "글 쓸 에너지가 없다"고 말한 것에 **과도하게 반응**하면, 다음과 같은 논리적 함정에 빠진다:

```
"환자가 글 쓸 힘이 없다"
  → "그러면 글 안 써도 되게 하자"
    → "이모지만 탭하면 되게 하자"
      → 가치 사슬 파괴
```

올바른 논리 흐름:

```
"환자가 글 쓸 힘이 없다"
  → "그러면 글을 안 쓰는 것 자체를 감지하자"
    → "미기록 패턴을 분석하자"
      → 시계열 패턴 감지기
```

## 3. 기능 제안 시 자문 체크리스트

새로운 기능을 제안하기 전에 반드시 아래 질문에 답하라:

### 체크리스트

| #   | 질문                                                                                                 | 기대 답변                                    |
| :-- | :--------------------------------------------------------------------------------------------------- | :------------------------------------------- |
| 1   | **이 기능이 사용자의 텍스트 입력을 대체하거나 생략하게 만드는가?**                                   | "아니오"여야 한다                            |
| 2   | **이 기능을 추가하면 AI가 분석할 데이터의 품질이 유지되거나 향상되는가?**                            | "유지" 또는 "향상"이어야 한다                |
| 3   | **이 기능은 기존 가치 사슬 위에 새로운 능력을 "얹는" 것인가, 기존 사슬의 일부를 "대체하는" 것인가?** | "얹는 것"이어야 한다                         |
| 4   | **이 기능이 없어도 기존 시스템은 잘 작동하는가?**                                                    | 이것만으로는 판단 불가. 아래 5번과 함께 본다 |
| 5   | **이 기능을 추가하면 시스템이 "할 수 없던 일"을 할 수 있게 되는가?**                                 | "예"여야 의미 있다                           |

### 판정 기준

- **1번에서 "예"가 나오면**: 즉시 기각. 어떤 편의성 논리도 가치 사슬 파괴를 정당화하지 못한다.
- **3번에서 "대체"가 나오면**: 기존 기능보다 명확히 우월한 근거가 없는 한 기각.
- **5번에서 "아니오"가 나오면**: 그것은 "기능"이 아니라 "UI 작업"이다. 핵심 제안이 아닌 부수 작업으로 분류하라.

## 4. 사례 연구: 시계열 패턴 감지기는 왜 통과하는가

### 체크리스트 적용

| #   | 질문                                        | 시계열 패턴 감지기의 답변                                                      |
| :-- | :------------------------------------------ | :----------------------------------------------------------------------------- |
| 1   | 텍스트 입력을 대체하는가?                   | **아니오.** 사용자는 여전히 글을 쓴다.                                         |
| 2   | 데이터 품질이 유지되는가?                   | **향상.** 단일 시점 데이터에 시간 축이 추가된다.                               |
| 3   | 얹는 것인가, 대체하는 것인가?               | **얹는 것.** 기존 분석 위에 추세 분석 레이어 추가.                             |
| 4   | 없어도 기존 시스템은 작동하는가?            | 예. 현재도 작동한다.                                                           |
| 5   | 시스템이 할 수 없던 일을 할 수 있게 되는가? | **예.** "침묵(미기록)"과 "행동 변화(시간대, 빈도 추세)"를 감지할 수 있게 된다. |

### 시계열 패턴 감지기의 본질

```
기존 시스템이 보는 것:        시계열이 추가로 보는 것:
━━━━━━━━━━━━━━━━━━━           ━━━━━━━━━━━━━━━━━━━━━

[오늘 쓴 텍스트] → 분석        월~수: 기록 있음
                               목~일: 기록 없음 → 🚨 미기록 패턴

[오늘 마음 온도 45°] → 주의    78° → 65° → 52° → 45° → 38°
                               → 🚨 5일 연속 하락 추세

[새벽 3시 기록] → 그냥 저장    평소 오후 9시 → 새벽 1시 → 2시 → 3시
                               → 🚨 수면 패턴 이상 감지
```

- 기존 가치 사슬을 **한 글자도 건드리지 않는다**
- 기존 데이터의 **시간적 흐름**이라는 새로운 차원을 읽는다
- **사용자가 아무것도 안 해도** 작동한다 (침묵 자체가 신호)

## 5. "기능 목록 부풀리기" 경계

기능을 제안할 때 흔히 발생하는 오류:

### ❌ 나쁜 제안 방식

```
"3계층 안전망 시스템을 제안합니다!"
 ├─ Layer 1: 이모지 1탭 + 음성 기록 + 예/아니오 응답
 ├─ Layer 2: 키워드 감지 + 패턴 분석 + 온도 급락 감지
 └─ Layer 3: SOS 버튼 + 1393 연결 + 보호자 알림

→ 13가지 항목을 3계층으로 포장 → "거대한 새 기능"처럼 보임
→ 실제로는 이미 구현된 것 + UI 작업 + 진짜 새 기능 1개
```

### ✅ 올바른 제안 방식

```
"시계열 패턴 감지기 1개를 제안합니다."
- 이미 있는 것: 키워드 감지, SOS, 보호자 알림, 1393 연결 (건드리지 않음)
- 새로 만들 것: 시계열 분석 엔진 (기존 데이터 위에 얹음)
- UI 후속 작업: 필요 시 대시보드 시각화 보강 (부수 작업)
```

## 6. 기존 기능과 새 기능의 관계 원칙

```
                    새 기능
                      │
                      │ "얹는다" (O)
                      ▼
 ┌─────────────────────────────────┐
 │   기존 가치 사슬 (건드리지 않음)   │
 │                                   │
 │  사용자 글쓰기 → AI 분석 →        │
 │  마음 온도 → 대시보드 → 알림      │
 └─────────────────────────────────┘

                      ✕ "대체한다" (X)
                      ✕ "생략한다" (X)
                      ✕ "단순화한다" (X)
```

**새 기능은 기존 가치 사슬의 위에** 놓여야 한다. 기존 사슬의 중간에 끼어들어 무언가를 빼거나 바꾸는 것은 "개선"이 아니라 "파괴"다.

## 7. 요약: 마음온 기능 제안의 황금률

> **"사용자의 텍스트를 지키면서, 그 텍스트에서 더 많은 것을 읽을 수 있게 만드는가?"**

- **"예"라면**: 제안하라.
- **"아니오"라면**: 아무리 편리해 보여도, 아무리 이해관계자가 원해도, **기각하라.**
